# Naruto Codex Deliberation Council 1.0.1

Release date: 2026-07-19

Version `1.0.1` is a security, runtime-contract, validation, and release-evidence
hardening update. It preserves the exact `$naruto` trigger, six runtime IDs,
four method IDs, one Naruto actor identity, Tsunade parent-Hokage role, and
standalone package layout introduced by `1.0.0`.

## Security And Installer Hardening

The installer now fails before writes on unknown, duplicated, missing-value,
or scope-incompatible options. Project targets must be absolute and user scope
does not accept `--target`.

Source files must exactly match package metadata and checksums. Destination
ancestor, directory, nested, final-component, and dangling symlinks are
rejected, as are source and destination hard links. Source bytes are retained
from their verified file descriptors and are not reopened during copying.
Unexpected files in the package-owned Naruto skill directory block
installation. Target validation runs from a private verified validator copy,
never JavaScript reopened from the mutable target, before status `installed`.

The migration regression uses the complete historical `0.4.0` inventory from
commit `42cab91`: 23 package files are replaced, 10 remain unchanged, and none
are synthesized as new paths before final installed-skill validation.

The installer remains sequential rather than transactional. A filesystem or
final-validation failure can leave a partial set of verified files. This
release does not claim automatic rollback or uninstall. Node has no portable
`openat`-style API, so the target must remain quiescent and exclusively
controlled by the installer user; do not run the installer elevated.

## Runtime Contract Hardening

- All child spawns use fresh context rather than inherited parent conversation
  history.
- The four Naruto candidate instances are opened before any candidate result is
  collected.
- Live preflight checks trusted project configuration, all six exact runtime
  IDs, peak capacity, effective parent and child `read-only` plus `never`
  approval policy, and exact-target follow-up support.
- Same-thread continuity is proved by successful follow-up receipts to the
  exact targets returned at spawn. Model-visible opaque-handle hashes are not
  required or invented.
- The agent manifest schema advances from `3` to `4` to encode host preflight,
  parent-owned orchestration state, and the receipt-based continuation contract.
- Protocol artifacts remain parent-owned orchestration state in run context.
  Read-only children do not persist them to the filesystem.
- Missing or ambiguous host capability returns `blocked`.

Profile read-only and never-approval settings remain defaults. Parent overrides
and host policy govern effective permissions, which is why live preflight is
mandatory.

## Static Validator And Receipt-contract Hardening

The existing dependency-free repository self-check covers the manifest-v4
host preflight, parent-owned orchestration state, fixed runtime and method IDs,
and same-target receipt contract fail-closed when static files or fixtures
disagree. Static classifiers now reject missing supervision, training,
guidance, attestation, and final-QA fields. Profile TOML and agent-card YAML are
parsed strictly, and checkpoint fixtures bind run, task, phase, and snapshot
projections. This remains a package self-check, not a parser or proof of an
arbitrary live run. A complete recorded-run validator is deferred to `1.1.0`.

## CI And Repository Hardening

- Hosted CI covers Ubuntu with Node 22 and 24 plus macOS and Windows with Node
  24.
- `actions/checkout` and `actions/setup-node` are pinned to full commit SHAs;
  checkout credentials are not persisted.
- Workflow concurrency and timeouts bound duplicate and stalled runs.
- Dependabot tracks GitHub Actions updates, and issue, security-contact,
  feature-request, and pull-request templates standardize public intake.
- The dangling-destination lifecycle fixture uses a removed directory-junction
  target on Windows, matching NTFS junction semantics without weakening the
  tested fail-closed behavior.
- Release-readiness normalizes repository inventory paths to forward slashes,
  preserving exact self-exclusions and public-claim scans on Windows.

## Acceptance Boundary

Static package and isolated installer tests must pass for the exact release
checkout. Ubuntu Node 22/24, macOS Node 24, and Windows Node 24 CI for the
exact `v1.0.1` tag must also pass before release acceptance.

These gates do not prove that a target Codex host discovers all six profiles,
applies effective child permissions, supplies required capacity, isolates
child context, or supports same-target revision. Until a target-build run
records that evidence, live runtime status remains `NOT PROVEN`; an attempted
run without the capabilities returns `blocked`.

This release does not bundle a host orchestration adapter or live event log and
does not claim measured behavioral diversity, quality improvement, latency,
token, or cost results. Those claims require execution of the documented
benchmark plan and independent review of its retained evidence package.
